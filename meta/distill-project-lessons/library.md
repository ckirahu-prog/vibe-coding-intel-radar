# 泛用 Vibe Coding 经验库

跨项目记忆。每条都是可迁移的原则，**不含**任何项目名、文件路径、数据源名、厂商名或一次性事故。
开新项目前可当预检清单读；每次复盘后（去重）追加。

条目格式：

```
## <精简原则标题>
原则：<一句话>
判断时机：<何时适用 / 如何检查>
行动：<可执行清单>
```

---

## 先定义交付物，再搭系统
原则：多数额外复杂度来自一上来就搭架构，而不是先想清楚产出。
判断时机：如果你说不清用户最终会看到什么（页面、报告、邮件、CLI 输出、demo），就还不该动手建。
行动：
- 先把最终交付物做成 mock。
- 定清楚用户、频率、成功标准。
- 从 mock 倒推数据、代码、prompt、自动化。

## prompt 是可执行的规格，不是许愿
原则：可靠的 agent 产出来自规格，而不是「做得好一点」这种愿望。
判断时机：弱 prompt 至少缺一项：输入范围、输出路径、格式、禁止动作、验收检查、失败处理。
行动：
- 写明角色、输入范围、输出位置/格式。
- 列出禁止动作，以及是否允许用工具/联网。
- 加一份 agent 必须自检的验收清单。
- 定义证据缺失或产出会为空时怎么办。

## 给易幻觉环节设证据闸门
原则：凡是模型能「脑补」的内容，都要配可验证的来源要求。
判断时机：关于用户需求、市场、收入、性能、安全、竞品、反馈的论断都是高风险。
行动：
- 要求引用、链接、日志、指标或可复现步骤。
- 没来源就写「未知」，不要编造。
- 推断出来的内容必须显式标注为推断。

## 允许空输出
原则：强行凑够固定数量的发现会摧毁可信度。
判断时机：任何「必须有 N 条」的板块都会被弱内容填满。
行动：
- 允许「本轮无」并写明原因。
- 弱信号单独放进低置信度区。
- 单来源条目标注「待交叉验证」。

## 限制工具，而非禁用或放任
原则：完全禁用工具会饿死任务；完全放开工具会失控。
判断时机：当部分步骤确实需要外部查询、另一部分必须纯本地时适用。
行动：
- 指明哪些步骤可用工具、哪些只读。
- 按类别给调用次数设上限，留预算。
- 要求审计「取了什么、为什么取」。

## 自动化成功 ≠ 产品成功
原则：跑出绿色不代表用户拿到了可用结果。
判断时机：存在多个独立检查点：产出文件、存到正确位置、触发下游、用户收到、可读、成本在预算内。
行动：
- 目标产物缺失时大声失败（非零退出码）。
- 打印读了哪些输入、产出了什么。
- 逐个检查点单独验证，而非只看「没报错」。

## 先跑通最窄回路
原则：先证明「输入 → 用户可见产出」，再加复杂度。
判断时机：如果在核心回路跑通前就有定时、邮件、数据库、多源、华丽 UI，调试会变模糊。
行动：
- 先交付最小端到端路径。
- 跑通后再加自动化、触达、监控。

## 生产 prompt 要稳定，知识就近内联
原则：运行时加载大量外部 skill/文档会让成本和行为都不稳定。
判断时机：如果每次运行都拉大段参考资料，产出会漂移、token 会暴涨。
行动：
- 大量参考资料只在设计阶段用。
- 把一份简短固定的速查表内联进生产 prompt。
- 让运行时输入尽量小且确定。

## 用户可见面是产品的一部分
原则：在源格式里看起来对，不代表在用户那端渲染对。
判断时机：在邮件、聊天、IM 等客户端查看的产出，常被剥样式或错渲染非标准格式。
行动：
- 在真实客户端实测，而非只看源格式。
- 关键样式内联；别依赖会被剥掉的头部样式。
- 源格式保持标准，让渲染器正常工作。

## 每个问题都归到一层
原则：分层归因能把一次性 bug 变成可迁移经验。
判断时机：发生过修复或返工、想提炼可复用结论时。
行动：
- 给每个问题打标签：需求 / 数据 / prompt / 执行 / 集成 / 呈现 / 运维。
- 修那一层，而不是只修表象。
- 把所属层记进经验里，让它能泛化。

## 钉死 agent 自动化的落点
原则：自治 agent 可能提交到旁支分支，悄悄跳过下游触发。
判断时机：任何自动化必须触发后续任务（邮件、部署、构建）时适用。
行动：
- 在指令里写明目标分支与远程。
- 定义下游任务监视的是哪条路径。
- 跑完后核实产物落在触发器期望的位置。

## 手动/高风险触发默认选最安全项
原则：误触发的手动运行应当什么坏事都不做，而不是执行最大范围动作。
判断时机：任何带模式/范围参数的手动或 CI 触发，或任何破坏性默认值。
行动：
- 默认设为最窄、最安全的范围（单一产物、dry-run、仅最新）。
- 宽范围或破坏性模式必须显式选择才生效。
- 在 UI/文档里写明默认值，让一键点过去的结果可预期。

## 自动提交的 bot 会让共享分支分叉
原则：定时任务往共享分支提交，会反复跑到你本地历史前面。
判断时机：任何由 CI/cron 推提交的仓库（数据快照、版本号、生成文件）。
行动：
- 动手前先同步，推之前再同步一次。
- 历史纠缠时，把要的那个提交 cherry-pick 到干净基底，而非整条 rebase。
- hard reset 后可能丢失 author identity，提交前先显式设置。

## 本地环境不可靠时，留一条 CI 运行路径
原则：别假设本地环境可复现；要有一条远程的运行与验证途径。
判断时机：本地工具链被禁、不在 PATH、或出现操作系统相关故障时。
行动：
- 与其和本地环境较劲，不如在 CI 里跑通并验证代码路径。
- 本地只做编辑，让可靠的运行器去执行。
- 把可运行命令固化进 CI，任何地方都能复现。
- 若本地到远程仓库的网络不稳定，把 write+push 交给云端 agent，本地只 pull 验收。

## Push 触发的下游要看整次 push，不能只看 HEAD
原则：一次 push 可能含多个 commit；path filter 会因中间某个 commit 触发，但 HEAD 未必含目标文件。
判断时机：下游 job 在 push 时扫描变更文件（发信、部署、构建镜像等）。
行动：
- 用 `event.before..event.after`（或等价整次 push diff）找变更，而非只看 HEAD commit。
- checkout 时 `fetch-depth: 0`，确保能对比 push 前后。
- 检测失败时回退到「发/部署最新产物」，避免误报 exit 1。

## 产物可能提前产出时，加定时兜底触达
原则：若交付依赖 push 触发，而产物可能在计划日前就生成（试跑、重跑），用户会在「该收的日子」收不到。
判断时机：低频产物（周报、月报）允许提前生成，但用户期望固定时间收到。
行动：
- 在预期送达时间后加一条 cron 兜底（发最新产物）。
- 手动/试跑与正式送达解耦：试跑不保证触达，正式 cron 保证触达。
- 文档写清「什么时候该收到」，避免用户误判为故障。
- 若本地到代码托管网络不稳定，把 commit+push 交给云端 agent，本地以 pull 同步为主。

## push 触发的下游要看整次 push 范围
原则：path filter 可能因 push 内任一 commit 触发，但 HEAD 未必含目标文件。
判断时机：下游 job 在 push 后扫描变更文件来决定是否执行（发信、部署、构建通知等）。
行动：
- 用 push 的 before..after 整段 diff 找目标产物，不要只看 HEAD 单个 commit。
- checkout 需要足够 depth（或 fetch-depth: 0）以对比 push 前后。
- 扫不到时回退到「取最新产物」并告警，而非直接 exit 1。

## 产物可能提前产出时，加定时兜底触达
原则：若交付依赖 push 触发，试跑或提前生成会让「正日子」没有新 push，用户收不到。
判断时机：允许手动/试跑提前产出最终交付物，且触达链是 push-triggered。
行动：
- 在预期送达时间后加一条 cron 兜底（发最新产物）。
- 手动触发与定时触发共用「取最新」逻辑，不要重复实现两套。
- 文档写清：平时不触达、只在特定节奏触达，避免用户误解。
